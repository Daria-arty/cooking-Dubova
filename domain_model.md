Вот результат:# Доменная модель (контекст клиентского мобильного приложения)

> Источник: бриф «Кулинарная студия» + уточнения R-004, R-008, R-015, R-027, R-028.  

> Ограничение: описание только роли «Клиент»; бэкенд — источник истины; клиент не создаёт слоты, программы и шефов.

---

## 1. Акторы (в контексте Клиента)

### 1.1. Клиент (основной актор клиентского приложения)

**Кто это:** Человек, который с телефона просматривает расписание кулинарных классов студии «Шеф-стол», выбирает свободный слот, создаёт бронь (в т.ч. выбор экипировки — своя или прокат), при необходимости отменяет бронь, получает уведомления (в т.ч. при отмене класса студией), после прошедшего класса может поставить оценку шефу.

**Цели в приложении:**

- Самостоятельно записаться на класс без переписки с Артёмом в WhatsApp.

- Видеть актуальные свободные места, программу, шефа и время начала.

- Управлять своей записью (отмена в рамках правил бэкенда).

- Быть проинформированным об отмене класса студией (форс-мажор: срыв поставки, поломка оборудования — из брифа).

**Не делает в клиентском приложении:** не формирует расписание, не назначает шефов, не отменяет класс целиком, не управляет прокатным фондом студии, не проводит оплату через приложение (в брифе оплата на старте — наличные/перевод, онлайн под вопросом).

---

### 1.2. Шеф (вторичный актор, вне интеракции в текущей поставке)

**Кто это:** Ведущий кулинарного класса; в студии около пяти шефов, в сезон — приглашённые (из брифа).

**Роль относительно клиентского приложения:** Объект выбора и отображения — клиент **видит**, какой шеф ведёт слот, и **может оценить** шефа после класса. Интерфейс шефа **не входит** в текущую поставку (R-028) — существует в отдельной инфраструктуре.

**Взаимодействие с клиентом:** Косвенное — через данные слота и форму оценки; прямого чата/контакта в брифе нет.

---

### 1.3. Организатор / студия (владелец — Артём) — вне клиентского UI

**Кто это:** Владелец кулинарной студии, операционно управляет расписанием, вместимостью (12 рабочих столов, до 8 человек на «сложных» классах), отменой классов при форс-мажоре.

**Роль относительно клиентского приложения:** Не является пользователем **данного** приложения (админка уже реализована в инфраструктуре, R-028). Влияет на клиента **через бэкенд**: формирует слоты, меняет статус брони на «Отменён студией» с причиной (R-008), обеспечивает push-уведомление.

**Для клиента проявляется как:** статусы и причины отмены, контактные данные студии, адрес студии (из API — R-015).

---

### 1.4. Система уведомлений (технический актор)

**Кто это:** Канал доставки push-уведомлений клиенту при отмене класса студией (R-008); в брифе также упомянуты напоминания о записи как опциональная идея.

**Роль:** Доставляет события, инициированные бэкендом/инфраструктурой; клиентское приложение регистрирует подписку и отображает payload (детали реализации push — на границе API и ОС, не в бизнес-логике бронирования).

---

## 2. Сущности

Для каждой сущности указан тип операции **с точки зрения клиентского приложения** (чтение с API vs мутация через API). Сущности, которых нет в брифе явно, но без которых сценарии не замкнуться, помечены как **вывод из сценария** — минимально, без расширения домена.

---

### 2.1. Слот (элемент расписания / кулинарный класс в расписании)

**Описание:** Единица расписания на конкретную дату и время начала: проведение группового кулинарного класса (~3 часа по брифу), с привязкой к программе, шефу и числу свободных рабочих мест. Горизонт планирования студии — неделя вперёд; в клиенте по умолчанию показываются ближайшие 7 дней (R-027).

**Атрибуты (по брифу и R-015):** время начала; программа (какое меню готовят); шеф; количество свободных рабочих мест; вместимость определяется бэкендом (12 столов студии, но на классах с духовкой и сложной техникой — не более 8 — правило на стороне бэкенда); адрес студии; прокатный тариф и свободный прокатный фонд (R-015) — в контексте слота/бронирования.

**Тип операции:** **Только чтение (приходит из бэкенда).**  

Клиент не создаёт, не редактирует и не удаляет слоты. Фильтр дат — параметр запроса, не изменение сущности.

**Ограничение (R-008):** На слот, отменённый студией, **повторная запись запрещена** — проверка на бэкенде; клиент не инициирует повторное бронирование на этот слот.

---

### 2.2. Программа (программа класса / меню)

**Описание:** Тип кулинарного занятия — например, «простая итальянская кухня для новичков» или более сложная программа для опытных (из брифа). В расписании отображается, **какое меню готовят** на конкретном слоте.

**Тип операции:** **Только чтение (приходит из бэкенда).**

---

### 2.3. Шеф

**Описание:** Человек, который ведёт класс на слоте. Клиент видит привязку шефа к слоту; после класса может поставить **оценку шефу** (из брифа).

**Тип операции:** **Только чтение (приходит из бэкенда)** — карточка/имя шефа в составе слота.  

Оценка шефа — отдельная мутация (см. «Оценка шефа»), не изменение сущности «Шеф» на клиенте.

---

### 2.4. Бронь (запись клиента на слот)

**Описание:** Факт записи клиента на конкретный слот. Центральная мутабельная сущность клиентского приложения. Заменяет ручную запись Артёма в Google-таблице/WhatsApp.

**Атрибуты (по брифу и R-008):** привязка к слоту и клиенту; выбор экипировки (своя / прокат); статус брони, в т.ч. **«Отменён студией»** с указанием **причины** (R-008) — бронь **не удаляется**, меняется статус; возможность отмены клиентом (правила — на бэкенде, в брифе акцент на поздней отмене за ~10 минут как операционной проблеме).

**Тип операции:** **Мутация (создаётся/меняется на клиенте и отправляется в API).**

- **Создание:** запись на свободный слот (атомарная проверка мест — R-004 на бэкенде).

- **Изменение статуса:** отмена клиентом (если разрешено API); отмена студией — инициируется инфраструктурой, клиент **получает обновлённое состояние** (read после push/sync).

- **Удаление:** явно не описано; при отмене студией бронь **сохраняется** со статусом (R-008) — не физическое удаление с точки зрения клиента.

---

### 2.5. Выбор экипировки (своя / прокат)

**Описание:** Решение клиента при бронировании: приходить **со своим фартуком и набором ножей** или взять **напрокат** (из брифа). Связано с прокатным тарифом и свободным прокатным фондом рабочих наборов (R-015).

**Тип операции:** **Мутация** — часть создания (или обновления, если API поддерживает) **брони**; отправляется в API.  

Доступность проката (**только чтение**): свободный фонд и тариф приходят из бэкенда; клиент не управляет складом проката.

---

### 2.6. Свободные места / вместимость слота

**Описание:** Число оставшихся рабочих мест на слоте; клиент должен **видеть свободные слоты** и записаться, пока места есть (из брифа). Актуальность и защита от двойной записи — на бэкенде (R-004).

**Тип операции:** **Только чтение (приходит из бэкенда)** в составе слота; уменьшение при успешной брони — следствие мутации «Бронь» на сервере, клиент обновляет отображение по ответу API.

---

### 2.7. Оценка шефа

**Описание:** Оценка, которую клиент **ставит шефу после класса** (из брифа); мотивация владельца — понимать, «кто как заходит людям», у звёздных шефов очередь.

**Тип операции:** **Мутация (создаётся на клиенте и отправляется в API)** — привязка к завершённой брони/прошедшему классу (детали — в контракте API).  

Просмотр ранее отправленной оценки — **только чтение**, если API отдаёт.

---

### 2.8. Аллергии и диетические ограничения

**Описание:** Информация, которую «хорошо бы заранее спрашивать», чтобы не узнавать на месте (из брифа); в брифе не зафиксирован момент сбора.

**Тип операции:** **Мутация** — если клиент заполняет поле (при брони или в профиле) и отправляет в API; иначе вне приложения.  

**Вывод из сценария** для замыкания потока; обязательность не задана в брифе.

---

### 2.9. Push-уведомление (событие для клиента)

**Описание:** Уведомление клиенту при отмене класса студией (R-008); в брифе также идея напоминаний о записи.

**Тип операции:**  

- **Доставка / payload:** инициируется бэкендом — для клиента по сути **только чтение** (отображение).  

- **Подписка на push (регистрация device token):** **Мутация** на клиенте → API (если предусмотрено контрактом).  

Клиент не создаёт бизнес-событие «отмена класса» — только реагирует.

---

### 2.10. Адрес студии / контакты студии

**Описание:** Адрес студии (лофт на бывшем заводе — из брифа); контакты для связи в брифе явно через WhatsApp сейчас, в приложении — не детализировано.

**Тип операции:** **Только чтение (приходит из бэкенда)** — R-015 упоминает адрес студии в данных API.

---

### 2.11. Клиентский профиль / учётная запись клиента

**Описание:** Не описан в брифе явно, но необходим для «мои брони», отмены, оценки и push (**вывод из сценария**, минимально).

**Тип операции:**  

- **Чтение:** данные профиля с API.  

- **Мутация:** регистрация/вход, обновление контактов — если предусмотрено API (бриф не детализирует).

---

### 2.12. Платёж / оплата класса

**Описание:** В брифе — наличные и перевод на карту; онлайн-оплата под вопросом на старте.

**Тип операции:** **Вне мутаций клиентского MVP по брифу** — не создаётся и не меняется в клиентском приложении, если онлайн-оплата не включена. Статус оплаты в брифе не описан.

---

### 2.13. Постоянный клиент / лояльность

**Описание:** «Постоянных клиентов много, отметить / дать что-то приятное» (из брифа) — без клиентских экранов в R-028.

**Тип операции:** **Не входит в ответственность клиентского приложения** текущей поставки; если данные появятся в API — максимум **только чтение** бейджа, но в брифе это не подтверждено → **не включаем в модель MVP**.

---

### 2.14. Причина отмены студией

**Описание:** Текстовое объяснение при статусе «Отменён студией» — срыв поставки продуктов, поломка оборудования, решение организатора (из брифа + R-008).

**Тип операции:** **Только чтение (приходит из бэкенда)** как часть состояния брони/слота; задаётся инфраструктурой, не клиентом.

## 3. Границы системы

### 3.1. Внутри ответственности клиентского мобильного приложения

| Область | Что делает клиент |

|--------|-------------------|

| **Просмотр расписания** | Показ слотов на 7 дней по умолчанию (R-027), расширение периода фильтром дат; empty state «Пока нет доступных классов». |

| **Отображение данных бэкенда** | Программа, меню, шеф, время начала, свободные места, адрес, прокатный тариф и остаток проката (R-015) — без локального «authoritative» хранения. |

| **Бронирование** | Отправка запроса на создание брони с выбором экипировки; обработка успеха и отказа (нет мест — R-004). |

| **Управление своими бронями** | Список броней клиента; отмена в рамках правил API; отображение статуса «Отменён студией» и **причины** (R-008). |

| **Ограничение повторной записи** | UI + запросы API: не предлагать/блокировать повторную запись на слот, отменённый студией (R-008). |

| **Оценка шефа** | Форма и отправка оценки после прошедшего класса (из брифа). |

| **Уведомления (клиентская часть)** | Регистрация на push, отображение уведомления об отмене студией (R-008); опционально напоминания — если включены в MVP. |

| **Сбор аллергий** | Ввод и отправка в API, если сценарий включён (из брифа — «хорошо бы заранее»). |

| **Обработка ошибок API** | Корректные сообщения при отказе бронирования, сетевых ошибках; повтор запроса по действию пользователя. |

| **Mobile UX** | Mobile-first веб-интерфейс для использования с телефона (из брифа). |

---

### 3.2. За границей клиентского приложения (бэкенд / существующая инфраструктура)

| Область | Где живёт | Комментарий |

|--------|-----------|-------------|

| **Формирование и редактирование расписания** | Бэкенд / админка | Слоты «на неделю вперёд», назначение шефа, программы — клиент только потребляет (R-028). |

| **Правила вместимости (12 / 8 мест)** | Бэкенд | «На классы с духовкой и сложной техникой больше 8 человек не берём» — расчёт лимита не на клиенте. |

| **Атомарность бронирования, «0 двойных броней»** | Бэкенд (R-004) | Транзакционность, SLA, маппинг на внутренние модели — вне скоупа клиента. |

| **Отмена класса целиком (форс-мажор)** | Инфраструктура / админка | Срыв поставки, поломка оборудования — класс отменяют целиком; клиент получает статус брони и push (R-008). |

| **Установка статуса «Отменён студией» и причины** | Бэкенд | Бронь не удаляется (R-008). |

| **Инициация push при отмене студией** | Бэкенд + push-провайдер | Клиент только подписчик. |

| **Управление шефами и программами** | Бэкенд | CRUD шефов и программ — не клиент. |

| **Управление прокатным фондом** | Бэкенд | Учёт свободных наборов для проката (R-015). |

| **Интерфейсы шефа и владельца** | Существующая инфраструктура (R-028) | «Мне самому нужен экран расписания» — уже есть, не в текущей поставке. |

| **Онлайн-оплата** | Не определено / вне MVP | Наличные и перевод — текущая реальность; интеграция платежей не подтверждена. |

| **Программа лояльности для постоянных клиентов** | Не определено / админка | «Отметить / дать что-то приятное» — не в клиентском MVP. |

| **Каноническая схема данных и миграции** | Контракт API (R-015) | Легаси-данных нет; клиент не дозаполняет историю. |

| **Гарантии доставки уведомлений, часовые пояса сервера** | Бэкенд / инфра | Клиент отображает время из контракта. |

---

### 3.3. Граница «контракт API» (R-004, R-015)

Клиентское приложение **полагается на ответы API** как на единственный источник истины по слотам, местам, статусам брони и запретам (повторная запись на отменённый слот, нет мест).  

Клиент **не дублирует** серверные бизнес-правила для authoritative решений; может дублировать их **в UI превентивно** (скрыть кнопку отмены, disabled «Записаться» при 0 мест), но финальное слово — за бэкендом.

## 4. Глоссарий (алфавитный, строго по брифу и уточнениям)

| Термин | Определение |

|--------|-------------|

| **Админка / администратор / владелец** | Интерфейс владельца (Артёма) для просмотра и ручной правки расписания; уже существует в инфраструктуре, **не входит** в текущую поставку клиентского приложения (R-028). |

| **Аллергии** | Диетические ограничения и аллергии участника; «хорошо бы заранее спрашивать», чтобы не узнавать уже на месте (из брифа). |

| **Бронь** | Запись клиента на конкретный кулинарный класс (слот). При отмене студией **не удаляется**, а переводится в статус «Отменён студией» (R-008). |

| **Вместимость / рабочие места** | Количество мест на класс; в студии **12 рабочих столов**, но на классах с духовкой и сложной техникой — **не более 8 человек** (из брифа). Клиент видит, **сколько мест свободно**. |

| **Клиент** | Человек, который записывается на кулинарные классы; основной пользователь **данного** приложения (R-028). |

| **Кулинарный класс** | Групповое занятие в студии, длится **около 3 часов**; разные **программы** для новичков и опытных (из брифа). |

| **Меню** | Состав блюд, которые готовят на занятии; в расписании указано, **какое меню готовят** на слоте (из брифа). |

| **Отмена клиентом** | Ситуация, когда записавшийся **отменяется**; если заранее — «ок, место освобождается»; если **за 10 минут до начала** — операционная проблема (продукты уже закуплены) (из брифа). |

| **Отменён студией** | Статус брони при отмене класса **организатором** или из-за **форс-мажора** (срыв поставки продуктов, поломка оборудования); с указанием **причины**; клиенту — **push-уведомление**; **повторная запись на этот слот запрещена** (R-008). |

| **Оценка шефа** | Оценка, которую клиент может **поставить шефу после класса** (из брифа). |

| **Постоянные клиенты** | Клиенты, которые регулярно ходят на классы; «было бы здорово их как-то отмечать / давать им что-то приятное» (из брифа) — без детализации в клиентском приложении. |

| **Программа** | Тип/тема занятия (например, простая итальянская кухня для новичков или более сложная программа); привязана к слоту в **расписании** (из брифа). |

| **Прокат (экипировки)** | Вариант для клиента без своего комплекта: **фартук и набор ножей «напрокат»** (из брифа). В API — **прокатный тариф** и **свободный прокатный фонд** рабочих наборов (R-015). |

| **Push-уведомление** | Уведомление клиенту, явно требуемое при статусе «Отменён студией» (R-008); в брифе также идея **напоминаний** о записи. |

| **Расписание** | Набор будущих классов; студия планирует **на неделю вперёд** — время начала, программа, шеф, свободные места (из брифа). В клиенте по умолчанию — **7 дней** (R-027). |

| **Своя экипировка** | Клиент приходит **со своим фартуком и набором ножей** (альтернатива прокату) (из брифа). |

| **Свободный слот** | Слот расписания, на который есть **свободные рабочие места** и доступна **запись** (из брифа: «клиент должен видеть свободные слоты и записаться»). |

| **Слот** | Элемент расписания: конкретное проведение класса в заданное **время начала** с конкретной **программой**, **шефом** и числом **свободных мест** (из брифа + R-027). |

| **Студия «Шеф-стол»** | Кулинарная студия Артёма в **лофте на бывшем заводе** (из брифа). |

| **Форс-мажор** | Обстоятельства, при которых **класс отменяют целиком**: сорвалась **поставка продуктов** или **сломалось оборудование**; нужно **предупредить всех** (из брифа) — на клиенте через статус брони и push (R-008). |

| **Шеф** | Ведущий кулинарного класса; в студии около **пяти** шефов, в сезон — **приглашённые** (из брифа). |

| **Экипировка** | **Фартук и набор ножей** — свои или прокат (из брифа). |-
