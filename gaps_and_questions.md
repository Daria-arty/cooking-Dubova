РЕЗУЛЬТАТ:# Пробелы, противоречия и вопросы заказчику (клиентское мобильное приложение)

> Контекст: роль «Клиент», бэкенд — black-box источник истины, контракт API — каноническая схема данных (R-015). Слоты, программы, шефы клиент не создаёт и не редактирует.

## 1. Идентификация и вход клиента

**Пробел:** В брифе не описано, как клиент «узнаётся» системой: регистрация, вход по телефону/e-mail, одноразовая ссылка или бронирование без аккаунта. Без этого неясны сценарии «мои брони», отмена, оценка шефа и push-уведомления (R-008).

**Вопрос Артёму:** Как клиент должен авторизоваться в мобильном приложении — обязательный аккаунт (телефон/e-mail) или достаточно указать контакты только при бронировании?

**MVP по умолчанию:** Обязательный минимальный вход по номеру телефона (SMS-код или mock-авторизация в учебном проекте). Все брони, отмены и оценки привязаны к `clientId` с бэкенда. Гостевое бронирование без входа — вне MVP.

---



## 2. Список «Мои брони» и история

**Пробел:** Артём хочет, чтобы люди «сами записывались», но явно не сказано, видит ли клиент список своих текущих и прошлыых записей, детали слота после бронирования и статусы (в т.ч. «Отменён студией» — R-008).

**Вопрос Артёму:** Нужен ли клиенту экран «Мои записи» с текущими и прошлыми бронями (дата, программа, шеф, статус)?

**MVP по умолчанию:** Да — отдельный раздел «Мои записи»: активные брони + архив (завершённые, отменённые клиентом, отменённые студией с причиной). Без этого невозможно осмысленно показать R-008 и сценарий оценки шефа.

---



## 3. Данные, обязательные при создании брони

**Пробел:** Не перечислено, какие поля клиент заполняет при записи, кроме выбора слота и экипировки (своя / прокат). Имя, телефон, количество мест, комментарий — не указаны. Неясно, можно ли записать несколько человек одной бронью или только одно рабочее место на клиента.

**Вопрос Артёму:** Одна бронь = одно рабочее место на одного человека, или клиент может забронировать несколько мест в одном классе одной операцией?

**MVP по умолчанию:** Одна бронь = одно место на одного клиента. Контактные данные берутся из профиля после входа; при бронировании клиент только выбирает слот и вариант экипировки. Повторная бронь на тот же слот тем же клиентом запрещена (если API вернёт ошибку — показываем сообщение бэкенда).

---



## 4. Выбор экипировки: своя или прокат

**Пробел:** Бриф требует выбор «со своим фартуком и набором ножей или напрокат». В уточнении (R-015) упомянуты прокатный тариф и свободный прокатный фонд, но не описаны: обязательность выбора, можно ли сменить до начала класса, что показывать при нулевом прокатном фонде, входит ли прокат в стоимость класса или это отдельная строка.

**Вопрос Артёму:** Выбор экипировки обязателен при каждой брони? Можно ли изменить «своя / прокат» после записи, и блокируется ли прокат, если свободных наборов не осталось?

**MVP по умолчанию:** Обязательный выбор при бронировании: «Своя экипировка» / «Прокат» (с отображением тарифа проката из API). Если `freeRentalStock = 0`, вариант «Прокат» недоступен с пояснением. Изменение выбора после бронирования — только через отмену и новую запись (если API не даёт PATCH) или через отдельный endpoint, если он есть в контракте; в MVP — без изменения после брони.

---



## 5. Правила отмены брони клиентом

**Пробел:** Артём описывает проблему отмены «за 10 минут до начала», но не формулирует правило для приложения: может ли клиент отменить сам, до какого момента, есть ли «мягкий» запрет или штраф, меняется ли статус брони при отмене клиентом (отличный от «Отменён студией» — R-008).

**Вопрос Артёму:** До какого времени до начала класса клиент может отменить запись в приложении без ограничений (например, за 24 часа / за 2 часа / до начала класса)?

**MVP по умолчанию:** Клиент может отменить бронь в приложении до **T−2 часов** от времени начала слота; позже — кнопка отмены скрыта, показывается текст: «Отмена возможна не позднее чем за 2 часа. Свяжитесь со студией: [контакты из API]». Порог 10 минут из брифа — операционная боль студии, в MVP клиенту показываем более ранний дедлайн, чтобы снизить поздние отмены. Статус при отмене клиентом — «Отменена клиентом» (если API отдаёт); место освобождается на стороне бэкенда.

---



## 6. Противоречие: «ограничить позднюю отмену, но не злить клиентов»

**Пробел:** Нет согласованной UX-политики: жёсткий запрет, предупреждение, контакт с администратором, условная отмена. Только клиентское приложение не может «наказать» без правила с бэкенда.

**Вопрос Артёму:** При попытке поздней отмены что должен видеть клиент — полный запрет, предупреждение с просьбой написать в WhatsApp, или возможность отмены с пометкой «поздняя отмена» для студии?

**MVP по умолчанию:** Жёсткий запрет отмены в UI после дедлайна (см. п. 5) + кнопка «Связаться со студией» (tel:/ссылка из API). Без штрафов и без «мягкой» отмены в MVP.

## 7. Оплата

**Пробел:** Артём «хотел бы сразу», но «не уверен, потянем ли онлайн-оплату на старте»; сейчас наличные и перевод на карту. Неясно, есть ли в клиентском приложении шаг оплаты, статус «оплачено / не оплачено», блокировка брони без оплаты.

**Вопрос Артёму:** В первой версии приложения клиент оплачивает класс только на месте/переводом вне приложения, или нужен экран оплаты в приложении?

**MVP по умолчанию:** Оплата **вне** клиентского приложения. После успешной брони — экран подтверждения с текстом: «Оплата при посещении или переводом по реквизитам студии» (реквизиты/инструкция — read-only из API, если поле есть; иначе статичный текст в конфиге). Поля payment в UI нет.

## 8. Аллергии и диетические ограничения

**Пробел:** «Хорошо бы заранее спрашивать про аллергии» — не указано: когда (при бронировании / за N дней до класса / в профиле), обязательность, формат (свободный текст / чеклист), можно ли редактировать после отправки.

**Вопрос Артёму:** Аллергии клиент указывает один раз в профиле, при каждой брони или отдельным шагом за день до класса? Поле обязательное?

**MVP по умолчанию:** Необязательное текстовое поле «Аллергии и ограничения» **при создании брони** (и опционально сохранение в профиле для подстановки в следующую бронь). Напоминание заполнить аллергии — in-app баннер за 24 часа до класса, если поле пустое; push — только если уже в scope уведомлений (см. п. 9).

---



## 9. Уведомления (напоминания и push)

**Пробел:** Напоминания о записи — «может, надо» (не решено). Push явно упомянут только для отмены класса студией (R-008). Не описаны: канал (PWA push / SMS), запрос разрешений, какие события покрывать в MVP.

**Вопрос Артёму:** Какие push-уведомления обязательны в первой версии: только отмена студией (R-008), или ещё напоминание за день/час до класса?

**MVP по умолчанию:** Push **обязательно** для «Отменён студией» (R-008). Напоминание за **24 часа** до начала класса — одно push на активную бронь. Запрос разрешения на push при первом входе или после первой успешной брони. SMS и e-mail — вне MVP.

---



## 10. Оценка шефа после класса

**Пробел:** «Поставить оценку шефу» — не определены шкала (1–5 / 1–10), текстовый отзыв, окно доступности (сразу после окончания / в течение N дней), одна оценка на бронь, можно ли менять оценку, что если класс «Отменён студией».

**Вопрос Артёму:** Оценка — только числовая (например, 1–5 звёзд) или число плюс короткий комментарий? В течение скольких дней после класса клиент может оценить?

**MVP по умолчанию:** Оценка **1–5** без обязательного текста; доступна для броней в статусе «Завершён» (или аналог с бэкенда) в течение **7 дней** после даты слота; одна оценка на бронь, без редактирования. Для «Отменён студией» и отменённых клиентом — оценка недоступна.

---



## 11. Отмена класса студией (форс-мажор) — детали UX

**Пробел (частично закрыт R-008):** Бронь не удаляется, статус «Отменён студией», причина, push, повторная запись на тот же слот запрещена. Неясно: показывается ли отменённый слот в общем расписании, нужен ли клиенту CTA «Выбрать другой класс», что делать с уже выбранным прокатом/оплатой (вне приложения).

**Вопрос Артёму:** Когда класс отменён студией, клиент должен видеть только свою бронь со статусом и причиной, или также карточку отменённого слота в общем списке (с пометкой «отменён»)?

**MVP по умолчанию:** В «Мои записи» — бронь со статусом «Отменён студией» и `cancellationReason` из API. В общем списке слотов отменённые студией слоты **не показываются** (или API не отдаёт их в выборку — клиент следует контракту). Кнопка «Подобрать другой класс» ведёт на список доступных слотов. Повторная запись на тот же `slotId` — UI блокирует + API отклоняет.

## 12. Фильтр дат и горизонт расписания

**Пробел (частично закрыт R-027):** По умолчанию 7 дней; более длинный период — «через фильтр дат». Не указаны максимальный диапазон фильтра, шаг (день / неделя), поведение при пустом периоде вне дефолтных 7 дней.

**Вопрос Артёму:** Насколько далеко клиент может смотреть расписание через фильтр — на 4 недели, на весь сезон, или только на те даты, которые уже есть в бэкенде?

**MVP по умолчанию:** Дефолт — ближайшие 7 дней (R-027). Фильтр: выбор произвольного диапазона **до 30 дней вперёд**; API запрашивается с `dateFrom` / `dateTo`. Если слотов нет — empty state «Пока нет доступных классов на выбранные даты».

---



## 13. Содержимое карточки слота в списке и деталях

**Пробел:** Из брифа известны: время начала, программа/меню, шеф, свободные места, длительность «где-то 3 часа», адрес студии (R-015). Неясно: показывается ли время окончания, уровень сложности программы, фото, описание «итальянская для новичков», лимит 8 vs 12 мест — нужно ли объяснять клиенту, почему мест меньше.

**Вопрос Артёму:** В карточке класса клиенту достаточно названия программы, даты/времени, шефа и числа свободных мест, или нужны отдельно уровень сложности и краткое описание меню?

**MVP по умолчанию:** Карточка слота: программа (название + краткое меню из API), дата и **время начала**, длительность (поле из API или фиксированные «~3 ч» если API отдаёт `durationMinutes`), шеф (имя), **свободные места / вместимость**, адрес студии (read-only, один раз в шапке или в деталях). Лимит 8/12 не объясняем — только цифра с бэкенда. Фото — только если есть URL в API.

---



## 14. Постоянные клиенты / лояльность

**Пробел:** «Отмечать постоянных / давать что-то приятное» — не определено, видит ли это клиент в приложении (бейдж, скидка, накопительная программа) или это только для админки Артёма.

**Вопрос Артёму:** Функции для постоянных клиентов должны быть видны в клиентском приложении в первой версии, или это внутренний инструмент студии?

**MVP по умолчанию:** **Вне MVP** клиентского приложения. Клиент не видит статус «постоянный»; логика на стороне существующей инфраструктуры/админки. В приложении — нейтральный UI для всех.

---



## 15. Контакты студии и связь при проблемах

**Пробел:** Сейчас запись через WhatsApp; в приложении не указаны телефон, мессенджеры, e-mail для связи при поздней отмене, форс-мажоре, вопросах по оплате.

**Вопрос Артёму:** Какие контакты студии должны быть в приложении (телефон, WhatsApp, адрес) и на каких экранах?

**MVP по умолчанию:** Блок «Контакты» в профиле/настройках: телефон (click-to-call), адрес студии (из API), опционально ссылка WhatsApp — все поля read-only с бэкенда. Дублирование на экране «не удалось отменить / ошибка бронирования».

---



## 16. Поведение при конкурентной брони (0 мест)

**Пробел (частично закрыт R-004):** Гарантия от двойных броней на бэкенде; клиент должен обрабатывать отказ. Не описан точный UX: обновление списка, сообщение, предложение другого слота.

**Вопрос Артёму:** При ошибке «мест больше нет» достаточно сообщения об ошибке, или нужно автоматически обновить список и подсветить другие слоты той же программы?

**MVP по умолчанию:** После ошибки API — toast/модал «Места только что закончились», автоматическое **обновление списка слотов** и снятие CTA «Записаться» на этом слоте `availableSeats = 0`). Без рекомендаций «похожих» слотов в MVP.

---



## 17. Часовой пояс и локаль

**Пробел:** Не указаны часовой пояс студии, формат даты/времени, язык интерфейса (бриф на русском).

**Вопрос Артёму:** Все времена в приложении — в часовом поясе студии (Москва/локальный город)? Интерфейс только на русском?

**MVP по умолчанию:** Все даты/время отображаются **как пришли из API** (бэкенд отдаёт в TZ студии, ISO 8601). UI только **русский**. Конвертация TZ на клиенте не делается.

---



## 18. Статусы брони — полный перечень

**Пробел:** Явно назван только «Отменён студией» (R-008). Не перечислены статусы активной брони, завершённого класса, отмены клиентом, возможного «ожидания» или «неявки».

**Вопрос Артёму:** Какие статусы брони клиент должен видеть в приложении (полный список)?

**MVP по умолчанию:** Отображаем все статусы, которые отдаёт API, с человекочитаемыми подписями минимум для: **Активна**, **Завершена**, **Отменена клиентом**, **Отменён студией** (+ причина). Логику переходов не дублируем — только отображение и действия, разрешённые API (отмена, оценка).

---



## 19. API-контракт как единственный источник — неизвестные endpoints

**Пробел (R-015):** Каноническая схема — контракт API, но сам контракт в брифе не приложен. Неизвестны точные названия полей, коды ошибок, пагинация слотов, идемпотентность создания брони.

**Вопрос Артёму (технический, через команду бэкенда):** Где зафиксирован актуальный OpenAPI/Swagger контракт клиентского API и какие операции уже реализованы на бэкенде?

**MVP по умолчанию:** Разработка клиента ведётся **strictly by contract** после получения OpenAPI; до этого — mock-сервер по минимальному набору: `GET /slots`, `GET /slots/{id}`, `POST /bookings`, `DELETE /bookings/{id}`, `GET /bookings/mine`, `POST /bookings/{id}/rating`, `GET /profile`, push-subscription endpoint если есть. Лишние поля не придумываем.

---



## 20. Противоречие скоупа: «приложение + API» vs «только клиент»

**Пробел:** В уточнении скоуп включает «(1) клиентское мобильное приложение, (2) API для него», при этом бэкенд уже существует. Для анализа **только клиента** граница: клиент потребляет API; реализация новых endpoint'ов — отдельная workstream, не описанная в брифе для клиентской логики.

**Вопрос Артёму:** Новый «API для клиента» — это тонкий BFF поверх существующего бэкенда или расширение текущего API? Кто owner контракта?

**MVP по умолчанию:** Считаем, что **контракт уже согласован** с существующим бэкендом; клиентское приложение не проектирует серверную транзакционность (R-004). Пробелы контракта закрываются mock'ами до интеграции.

## 21. PWA vs нативная оболочка

**Пробел:** «Мобильное веб-приложение» и использование «с телефона» не уточняют: installable PWA, адаптивный сайт в браузере, требования offline.

**Вопрос Артёму:** Достаточно ли адаптивного веб-приложения в браузере телефона с возможностью «Добавить на главный экран», или нужна отдельная обёртка?

**MVP по умолчанию:** **Mobile-first responsive web app** (PWA-lite: manifest + иконка; service worker только для push, если требуется). Offline-режим записи — не поддерживается; при отсутствии сети — экран «Нет соединения».
